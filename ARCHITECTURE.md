# Архитектура системы: генерация музыки с заданными структурными параметрами

---

## Общая идея

Цель — дать пользователю два рычага управления генерацией MIDI:

1. **Текстовый промт** — *какая* это музыка («нежная фортепианная мелодия в до-мажоре»)
2. **Структурный план** — *как* она устроена («куплет 16 тактов, припев 16 тактов, ...»)

Из этих двух входов система генерирует полный MIDI-трек с заданной формой и характером.

```
Текстовый промт          Сегментный план
      ↓                  "verse:16,chorus:16,..."
[Text2Prefix]                   ↓
      ↓                  [Seg2SSM]
  Prefix                      ↓
  (8–16 тактов MIDI)      Affinity SSM
        ↓_______________________|
              [SingLS / 3SING*]
                    ↓
              Полный MIDI-трек
```

---

## 1. SING — оригинальная модель

### Что это

SING (Similarity Incentivized Neural Generator) — авторегрессивная LSTM-модель, которая генерирует MIDI побитово. На каждом шаге модель смотрит не только на предыдущие ноты, но и на Self-Similarity Matrix (SSM) — матрицу структурного сходства трека.

### Как работает SSM

SSM строится по chroma-признакам: piano roll `[T, 128]` → chroma `[T, 12]` → cosine similarity `[T, T]`. Ячейка `SSM[i, j]` показывает насколько бит `i` похож на бит `j` по гармоническому содержанию. Высокое значение = похожие моменты (например, два куплета).

### Механизм attention в SING

На каждом шаге генерации `t` модель спрашивает SSM: «На какие прошлые биты `t` похож?» — и делает взвешенную сумму piano roll этих битов:

```
weights = sparsemax( SSM[t, :t] )          # [t] — разреженные веса
weight_vec = sum( prev_sequence * weights ) # [128] — взвешенный контекст
output = attention( [weight_vec, lstm_out] ) # [128] — финальные логиты
```

Sparsemax (в отличие от softmax) обнуляет несущественные веса — модель фокусируется на 2–5 наиболее похожих прошлых битах.

### Ключевое ограничение SING

Модель умеет *следовать* SSM во время генерации, но SSM передаётся как ground truth из реального трека. Для реального использования нужно уметь строить SSM из пользовательского описания структуры.

## 2. SingLS — улучшенные механизмы attention

Исследование трёх альтернативных способов использовать SSM внутри модели.

### 2.1 LSA (Latent Structured Attention)

Вместо прямого взвешивания piano roll — полноценный QKV-attention, где SSM используется как мультипликативная маска:

```
scores = Q @ K^T              # [T, T] — сырые оценки сходства
scores = scores * SSM         # SSM усиливает структурно обоснованные связи
output = softmax(scores) @ V  # взвешенный контекст
```

**Идея:** SSM не диктует веса напрямую, а только усиливает уже найденные моделью связи.

### 2.2 LSA_SB (Score Bias)

Более мягкий вариант: SSM добавляется как аддитивный бонус к логитам, а не маска:

```
scores = Q @ K^T + β · SSM    # β = 0.03
```

**Идея:** модель сохраняет собственное мнение об attention, SSM лишь слегка подталкивает к структурно правильным связям.

### 2.3 3SING* — HierarchicalGenerator

Самая сложная архитектура, показавшая лучшие результаты по IOU.

```
LSTM output [T, H]
      ↓
StructureModel(SSM) → struct embedding [T, H]   # глобальный структурный план
      ↓
mixed = lstm_out + α · struct                    # α = 0.05
      ↓
original_attention(mixed, prev_sequence, SSM)    # sparsemax attention поверх
      ↓
output [128]
```

StructureModel — это трансформер, обученный видеть всю SSM целиком и формировать глобальное представление о структуре. LSTM отвечает за локальную музыкальную связность, StructureModel — за глобальную форму.

### Что узнали из сравнения

| Модель           | IOU ↑      | MSE ↓      | ScaleCons ↑ | Вывод                                  | Чекпоинт                                                                               |
|------------------|------------|------------|-------------|----------------------------------------|----------------------------------------------------------------------------------------|
| SING (baseline)  | 0.2986     | 0.0800     | 0.5342      | хороший baseline                       | `data/meta_info/trained_original_combined/model_30_epochs.txt`                         |
| LSA              | 0.0959     | 0.0568     | 0.4561      | лучший MSE, слабее по IOU и гармонии   | `data/meta_info/trained_lsa_combined/model_30_epochs.txt`                              |
| **3SING***       | **0.3198** | **0.0692** | **0.5330**  | лучший баланс                          | `data/meta_info/trained_transformer_original_less_struct_combined/model_30_epochs.txt` |

LSA хорошо минимизирует MSE (структуры похожи точечно), но плохо воспроизводит крупные структурные паттерны (IOU) и даёт наименее тональную генерацию (ScaleCons=0.46). 3SING* с глобальным StructureModel показывает лучший баланс по всем трём метрикам.

> Метрики получены с `temperature=1.5` в сэмплировании. Без температуры модели коллапсировали на 1–2 нотах, и старые числа были недействительны.

## 3. Seg2SSM — из сегментного плана в SSM

### Задача

Пользователь задаёт структуру словами: `"verse:16,chorus:16,verse:16,chorus:16"`. Нужно превратить это в матрицу `SSM [T, T]`, которую модель поймёт как структурный prior.

### Affinity Matrix

Ключевая идея — не нейросеть, а явная матрица музыкального сходства между типами секций:

```
A (7×7):        intro  verse chorus bridge instr outro
         intro [  1.0   0.6   0.4    0.2   0.5   0.6 ]
         verse [  0.6   1.0   0.7    0.3   0.4   0.3 ]
        chorus [  0.4   0.7   1.0    0.4   0.5   0.5 ]
        bridge [  0.2   0.3   0.4    1.0   0.3   0.2 ]
         instr [  0.5   0.4   0.5    0.3   1.0   0.4 ]
         outro [  0.6   0.3   0.5    0.2   0.4   1.0 ]
```

Значения отражают музыкальную теорию: куплет и припев похожи (0.7), бридж контрастирует со всеми (0.2–0.4).

### Построение SSM

```python
bar_labels = expand(segment_plan)   # ["verse"]*16 + ["chorus"]*16 + ...
SSM[i,j]   = A[label_i, label_j]   # блочно-постоянная матрица
SSM        = gaussian_blur(SSM)     # сглаживание границ блоков
SSM        = resize(SSM, T_total)   # приведение к размеру трека
```

Gaussian smoothing отражает реальность: переходы между секциями не абруптные.

### Два режима

- **A_fixed** — задана из теории музыки, без обучения, воспроизводима. Основной режим.
- **A_empirical** — оценена из 422 треков SALAMI как среднее SSM по парам меток. Валидирует, что A_fixed соответствует реальным данным.

### Почему не нейросеть

Сегментный план содержит очень мало информации (7 типов секций). Нейросеть здесь избыточна и неинтерпретируема. Affinity Matrix — это компактная, объяснимая модель, которую можно показать на одном слайде и обосновать музыкально.

## 4. Text2Prefix — из текста в музыкальный фрагмент

### Задача

Пользователь описывает характер музыки текстом. Нужно превратить это в до 8–16 тактов piano roll `[T, 128]`, которые SingLS будет использовать как затравку (prefix).

### Пайплайн (text2midi — основной бэкенд)

```
текст → [T5Tokenizer] → [text2midi Transformer] → REMI-токены → piano roll
```

1. **text2midi** (`amaai-lab/text2midi`) — seq2seq трансформер (T5 encoder + REMI decoder), обученный на MidiCaps. Генерирует MIDI-токены напрямую, без аудио-транскрипции.

2. Токены декодируются через `miditok` → `symusic.Score` → `pretty_midi` → piano roll с max-pool до разрешения SingLS (1 фрейм/доля).

**Fallback:** если text2midi недоступен, пайплайн переключается на MusicGen + BasicPitch (аудио-бэкенд).

### Авто-тримминг prefix

text2midi может завершить генерацию раньше запрошенного числа тактов (ранний EOS). Вместо того чтобы паддить тишиной, пайплайн автоматически:

- детектирует последний активный бит в piano roll,
- обрезает `prefix_roll` до реального контента,
- пересчитывает `T_prefix = prefix_roll.shape[0]` и `T_total = T_prefix + T_gen`.

Красная линия на визуализации (`pipeline_overview.png`) сдвигается влево автоматически.

### Проблема OOD-prefix

text2midi производит piano roll, отличающийся по распределению от обучающих данных SingLS (MAESTRO/LMD). Если пропустить слишком много битов prefix через LSTM (`warmup_beats`), hidden state "заражается" OOD-паттернами и генерация коллапсирует.

**Решение:** `warmup_beats=4` (дефолт). Prefix задаёт тональный контекст через последние 4 бита LSTM, OOD-фреймы не попадают в attention context (`prefix_in_sequence=False`).

## 5. Финальный пайплайн

### Схема

```
python -m pipeline.generate \
    --model  <чекпоинт 3SING*> \
    --prompt "gentle piano melody in C major" \
    --segment "intro:8,verse:16,chorus:16,verse:16,chorus:16,outro:8" \
    --ssm_type affinity \
    --n_gen_bars 64 \
    --warmup_beats 4
```

```
prompt ──→ Text2Prefix ──→ prefix_roll [32, 128]
                                  │
segment ──→ Seg2SSM ──────→ SSM [256, 256]
                                  │
               └──── SingLS ─────┘
                       │
                    generate_from_prefix(model, prefix_roll, SSM, gen_len=224)
                       │
                    full_roll [256, 128]
                       │
                    → MIDI + piano roll PNG + SSM PNG
```

### Ablation study

Для оценки вклада каждого компонента предусмотрены 4 режима SSM:

| SSM Type    | Описание                              | Роль                                     |
|-------------|---------------------------------------|------------------------------------------|
| `none`      | нулевая SSM                           | baseline (модель без структурного prior) |
| `random`    | случайная симметричная                | нижняя граница                           |
| `affinity`  | AffinitySSM из сегментного плана      | целевой режим                            |
| `empirical` | матрица A, оценённая из SALAMI        | верхняя граница                          |

**Методология:** одинаковый prefix и модель, только тип SSM меняется. Референс для IOU/MSE — среднее SSM по 20 трекам из `combined_test.pt` (GT реальной музыки, не affinity SSM — иначе метрика зависела бы от условия).

**Запуск:**
```bash
python -m pipeline.ablation \
    --model data/meta_info/trained_transformer_original_less_struct_combined/model_30_epochs.txt \
    --prompt "gentle piano melody in C major, slow tempo" \
    --segment "intro:8,verse:16,chorus:16,verse:16,chorus:16,outro:8" \
    --tempo 90 \
    --n_gen_bars 64 \
    --out_dir outputs/ablation_gt_ref
```

Результаты сохраняются в `out_dir/`: MIDI для каждого условия, `ablation_overview.png`, `metrics.txt`.

**Результаты ablation (2026-04-04):**

| SSM Type    | IOU ↑  | MSE ↓   |
|-------------|--------|---------|
| `none`      | 0.0254 | 0.08348 |
| `random`    | 0.0284 | 0.07980 |
| `affinity`  | 0.0293 | 0.09154 |
| `empirical` | 0.0318 | 0.09110 |

IOU показывает ожидаемый порядок: `none < random < affinity < empirical`. Структурный prior работает — каждый следующий тип SSM даёт лучшее структурное сходство с реальной музыкой. MSE ведёт себя противоположно по той причине, что affinity/empirical SSM заставляет модель генерировать более повторяющиеся паттерны с более высокими значениями SSM в целом — это отдаляет от GT по MSE, но улучшает структурную точность по IOU.

Абсолютные значения IOU низкие (0.025–0.032) — это ожидаемо: референс представляет "среднюю реальную музыку", а не конкретный трек. Важен относительный порядок.

### Ключевые выводы диагностики

1. **Температура обязательна.** Без `temperature=1.5` в `topk_sample_one` авторегрессия коллапсирует на 1–2 нотах независимо от SSM и prefix. Это инференс-артефакт, не проблема модели.

2. **AffinitySSM не ломает генерацию.** Несмотря на то что обучающие SSM были органичными (chroma-cosine), блочно-постоянная AffinitySSM совместима с моделью — качество генерации сопоставимо с `ssm_type=none`.

3. **`warmup_beats=4` — оптимальный компромисс.** При 0 — генерация хаотична (нет музыкального контекста), при 10 — OOD-prefix искажает hidden state и вызывает коллапс.

4. **Концентрация нот в регистре prefix — не коллапс.** Модель когерентно продолжает тональный центр, заданный prefix. Реальный коллапс — это залипание независимо от входа.
