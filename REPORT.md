## 1. Подтверждение работоспособности структурного внимания: SSM-following

Ключевой вопрос работы — действительно ли подача Self-Similarity Matrix (SSM) в механизм внимания управляет структурой генерируемого трека, или модель её игнорирует.

Для проверки был проведён контролируемый эксперимент (N=100 случайных сегментных планов, 64 такта генерации на план):

1. По каждому плану строилась **целевая SSM** через `AffinitySSM` (аналитическая матрица на основе теории музыки).
2. Трек генерировался дважды — с этой SSM (`affinity`) и с нулевой SSM (`none`), при фиксированном одинаковом in-distribution prefix из тестового датасета.
3. Из сгенерированного piano roll вычислялась **реальная chroma-SSM**, которая сравнивалась с целевой по метрикам IOU и MSE.

**Результаты:**

| Условие   | IOU ↑             | MSE ↓           |
|-----------|-------------------|-----------------|
| affinity  | 0.0329 ± 0.0079   | 0.2877 ± 0.0415 |
| none      | 0.0280 ± 0.0081   | 0.2934 ± 0.0413 |
| **Δ**     | **+0.0049 ✓**     | **−0.0057 ✓**   |

Оба показателя значимо улучшились при подаче AffinitySSM. Вывод: **механизм SSM-внимания реально управляет структурой генерации** — модель производит треки, чья chroma-SSM ближе к заданному структурному плану.

## 2. Датасет

Модель обучена на объединённом датасете **combined** (~2600 треков), собранном из двух источников:

| Датасет                          | Треков   | Описание                                                    |
|----------------------------------|----------|-------------------------------------------------------------|
| MAESTRO                          | ~1300    | Записи академических фортепианных исполнений (MIDI + аудио) |
| LMD (Lakh MIDI Dataset, aligned) | ~1300    | Популярная и классическая музыка в формате MIDI             |

**Формат хранения:** каждый трек — кортеж `(piano_roll, tempo, num_beats)`, где piano_roll — бинарная матрица `[T, 128]`, T — количество битов с темпо-адаптивной частотой дискретизации (`fs = tempo / 60`). Диапазон валидных нот: 20–108 (клавиатура фортепиано).

Датасет разбит на `combined_train / combined_val / combined_test.pt` в пропорции примерно 80/10/10.

**Ограничения датасета.** 2600 треков — небольшая выборка для задачи с SSM-регуляризацией. Модель хорошо воспроизводит структурные паттерны, однако ограниченное разнообразие данных частично объясняет недостаточную «музыкальность» генерации.

**Датасет для Seg2SSM (SALAMI).** Для оценки эмпирической матрицы афинности использовался отдельный датасет SALAMI: 422 трека с ручной разметкой структурных секций (intro, verse, chorus, bridge, outro и др.). Из SALAMI извлекались chroma-признаки и bar-level SSM, по которым усреднялось сходство для каждой пары типов секций.

## 3. Seg2SSM — построение SSM из сегментного плана

### Задача

Пользователь задаёт структурный план в виде строки, например `"intro:8,verse:16,chorus:16,outro:8"`. Задача модуля — преобразовать этот план в матрицу структурного сходства SSM `[T, T]`, которая подаётся в механизм внимания генеративной модели.

### Принцип работы

Вместо нейросети используется **аналитическая матрица** `A` (7×7 по типам секций). Построение SSM:

```
segment_plan → bar_labels [T] → SSM[i,j] = A[label_i, label_j] → постобработка → resize [ssm_size × ssm_size]
```

Каждый бит генерации получает метку секции (verse, chorus и т.д.), затем SSM заполняется значением из матрицы `A` для пары меток. Результат — блочная матрица, отражающая музыкальную близость секций.

### Матрица A (A_fixed)

Задана вручную на основе теории музыки:

|        | intro | verse | chorus | bridge | instr | outro |
|--------|-------|-------|--------|--------|-------|-------|
| intro  | 1.00  | 0.60  | 0.40   | 0.20   | 0.50  | 0.60  |
| verse  | 0.60  | 1.00  | 0.70   | 0.30   | 0.40  | 0.30  |
| chorus | 0.40  | 0.70  | 1.00   | 0.40   | 0.50  | 0.50  |
| bridge | 0.20  | 0.30  | 0.40   | 1.00   | 0.30  | 0.20  |
| outro  | 0.60  | 0.30  | 0.50   | 0.20   | 0.40  | 1.00  |

Принципы: verse ↔ chorus высокое (0.70) — оба несут основной материал; bridge низкое со всеми (0.20–0.40) — контрастная секция; intro ↔ outro среднее (0.60) — обрамляют трек.

### Постобработка

Сырая блочная SSM имеет распределение `mean≈0.70, std≈0.22`, тогда как модель обучена на chroma-SSM с `mean≈0.50, std≈0.13`. Из-за несовпадения распределений sparsemax-внимание ведёт себя некорректно (насыщение весов).

Применяются три шага:
1. **Gaussian blur** (σ=1.5) — сглаживание ступенчатых границ блоков (≈1 такт)
2. **Симметричный шум** (std=0.04) — имитация естественной вариабельности chroma
3. **Linear rescale** → target mean=0.50, std=0.13 — приведение к обучающему распределению без разрушения блочной структуры


## 4. Модель 3SING* — архитектура, обучение, сравнение с SING

### Базовая модель: SING

Оригинальный SING (Similarity Incentivized Neural Generator) — авторегрессивная LSTM-модель для piano roll `[T, 128]`. Структурное внимание реализовано через sparsemax по строке SSM: на каждом шаге генерации модель взвешивает прошлые скрытые состояния пропорционально структурному сходству.

Функция потерь:
```
L = BCE(output, target) + λ · L2(SSM_generated, SSM_target)
```

### Расширение: 3SING*

3SING* (`HierarchicalGenerator`) добавляет к SING компонент глобального структурного планирования — `StructureTransformer`. Архитектура:

```
HierarchicalGenerator
├── generator = MusicGenerator (LSTM + ORIGINAL attention / sparsemax по SSM)
└── structure_model = StructureModel
    └── StructureTransformer (d_model=128, nhead=4, num_layers=2)
        ├── Row-wise Conv1d encoder: каждая строка SSM → вектор d_model
        ├── Learned positional embedding
        └── TransformerEncoder (2 слоя, 4 головы)
```

На каждом шаге генерации:
1. LSTM производит локальный выход `lstm_out [T, B, H]`
2. `StructureTransformer` кодирует текущую SSM → структурный контекст `struct [T, B, H]`
3. Смешивание: `mixed = lstm_out + α · struct`, где `α = 0.05` (фиксированный)
4. `_apply_attention(mixed, prev_sequence, SSM)` → финальный logit

Функция потерь расширена:
```
L = BCE + L2_ssm + β_struct · L_contrastive
β_struct = 0.03
```

**Ключевые технические решения и причины:**

| Решение                                | Почему                                                                                                  |
|----------------------------------------|---------------------------------------------------------------------------------------------------------|
| `α = 0.05` фиксированный               | Обучаемый α сходился к 0.0025 — структура игнорировалась                                                |
| `β_struct = 0.03` малый                | Большой β разрушал BCE-обучение нот; 0.03 работает как мягкая регуляризация                             |
| LSTM, не полный трансформер            | SSM уже покрывает долгосрочные зависимости; датасета (2600 треков) недостаточно для трансформера с нуля |

### Метрики на тестовой выборке combined (2600 треков)

| Модель                         | IOU ↑      | MSE ↓      | ScaleCons ↑   |
|--------------------------------|------------|------------|---------------|
| None (baseline, без генерации) | 0.0848     | 0.0645     | —             |
| SING (original)                | 0.2986     | 0.0800     | 0.5342        |
| **3SING\***                    | **0.3198** | **0.0692** | **0.5330**    |

3SING* превосходит оригинальный SING по всем ключевым метрикам: IOU +2.1 п.п. (+7%), MSE −0.0108 (−13.5%). Добавление `StructureTransformer` как слабой регуляризации (β=0.03) улучшает и структурное соответствие, и качество нот одновременно.

## 5. End-to-end пайплайн генерации

Реализован полный пайплайн, управляемый двумя входными сигналами:

```
Текстовый промт               Сегментный план
"gentle piano melody"         "intro:8,verse:16,chorus:16,..."
        ↓                              ↓
  [Text2Prefix]                  [Seg2SSM]
text2midi (LLaMA-based)        AffinitySSM
        ↓                              ↓
prefix_roll [T_prefix, 128]    SSM [T_total, T_total]
              ↘_______________↙
                  [3SING*]
           HierarchicalGenerator
                    ↓
         full_roll [T_total, 128]
                    ↓
         generated.mid  (только T_gen часть)
         prefix.mid     (отдельно)
         pipeline_overview.png
```

**Компоненты:**

- **Text2Prefix** (`Text2Prefix/text2prefix_midi.py`): text2midi на базе LLaMA генерирует MIDI напрямую без транскрипции. Преимущество перед MusicGen+BasicPitch: нет потерь при аудио→MIDI транскрипции, точный темп.
- **Seg2SSM** (`Seg2SSM/affinity_ssm.py`): аналитическое построение SSM из сегментного плана. Постобработка: Gaussian smoothing (σ=1.5) + noise (std=0.04) + rescale к обучающему распределению (mean≈0.50, std≈0.13). Rescale критичен: сырая блочная матрица имеет другое распределение, из-за чего sparsemax ведёт себя некорректно.

## 6. Текущие трудности

### 6.1 Музыкальное качество

Сгенерированные треки метрически соответствуют структурному плану (ΔIOU>0), однако **воспринимаются как набор случайных нот**, а не как пьеса. Выявленные причины:

**Out-of-distribution prefix.** Text2midi генерирует музыкально связный MIDI-фрагмент, но он может принадлежать другому ладу, темпу или фактуре, чем обучающий датасет (MAESTRO + LMD). LSTM прогревается на нехарактерных данных и не выходит в распределение тренировки даже после warmup_beats=4 тактов. Эта проблема является **главным источником деградации** качества демо.

**Отсутствие MIDI post-processing.** Модель генерирует бинарный piano roll с плоским velocity=80 и стаккато-нотами (без устойчивости звука). Добавление sustain pedal (CC64) и velocity variation ±10–15 значительно улучшило бы восприятие без изменения модели.

### 6.2 Открытые идеи к защите

- **Идея SSM-matching prefix:** вместо text2midi-prefix брать фрагмент из тестового датасета, чья SSM наиболее близка к целевой AffinitySSM. Это обеспечивает in-distribution прогрев LSTM при сохранении структурной согласованности.
- Улучшения pipeline (постпроцессинг MIDI, cherry-pick лучших генераций) для демонстрации максимально качественного результата на защите.
